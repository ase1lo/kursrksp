# Анализ предметной области

## Цель системы

CorpLink Messenger - корпоративный мессенджер для командной коммуникации внутри организации. Система объединяет каналы, сообщения, интеграции с внешними сервисами, управление пользователями, ролевую модель и аудит действий.

## Участники

- Администратор управляет пользователями, ролями, каналами, интеграциями и аудитом.
- Модератор создает рабочие каналы, подключает интеграции и модерирует сообщения.
- Участник читает доступные каналы и отправляет сообщения.
- Bot-пользователь публикует автоматические сообщения от CI, Git или webhook-интеграций.

## Основные сущности

- User: учетная запись, email, имя, роль, статус активности.
- Channel: рабочее пространство по теме, публичное или приватное.
- ChannelMember: связь пользователя и приватного канала.
- Message: сообщение в канале с автором и признаком редактирования.
- Integration: подключение внешнего сервиса к каналу.
- AuditEvent: журнал значимых действий.

## Бизнес-правила

- Роли ограничены списком `admin`, `moderator`, `member`, `bot`.
- Некорректная роль отклоняется на уровне API и покрыта тестами.
- Только `admin` управляет пользователями и читает аудит.
- `admin` и `moderator` управляют каналами и интеграциями.
- Приватные каналы читают только staff-пользователи или участники канала.
- Пароли не хранятся в открытом виде: используется PBKDF2-HMAC-SHA256.
- Авторизация реализована через подписанный HMAC bearer token.

## CRUD-операции

- Users: create, read, update, soft delete.
- Channels: create, read, update, delete.
- Messages: create, read, update, delete.
- Integrations: create, read, update, delete.
- AuditEvents: read.

## Риски и защита

- Неверные роли: строгая whitelist-валидация.
- Неверный JSON и невалидный UTF-8: контролируемый ответ 400.
- Подмена токена: HMAC-подпись и срок действия.
- Несанкционированный доступ: централизованные RBAC-проверки.
- Регрессии: 100% line coverage gate для backend-ядра и fuzzing-тесты.
